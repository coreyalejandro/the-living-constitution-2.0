# Portfolio Publication Rule

**Version:** 1.0.0
**Status:** Active — MANDATORY
**Last updated:** 2026-05-13

---

## Rule Statement

> The Portfolio is the public projection of the registry.
> It must not be hand-authored from scratch, vibes, or aspirational copy.
> Every claim on the Portfolio must be traceable to a registry entry.

---

## What the Portfolio is

The Portfolio is a public-facing site that displays truth-labeled module data
derived from the runtime registry. It is not a marketing site. It is not a
résumé supplement. It is an evidence surface.

The Portfolio serves one primary function:
**Show what is real, what is verified, and what the gaps are — without overclaiming.**

---

## Data source rule

The Portfolio must consume:
- registry/modules.registry.json
- registry/artifacts.registry.json
- registry/routes.registry.json
- PORTFOLIO_DATA.json (generated by scripts/generate-portfolio-data.mjs)

The Portfolio must NOT source content from:
- Hand-written copy files not backed by registry entries
- README.md files from external repos (unless quoted with attribution)
- Prior portfolio versions that were not registry-driven
- Aspirational descriptions of unverified modules

---

## Public display rules by truth_status

| truth_status | Allowed public display | Forbidden |
|---|---|---|
| working | "Working project" badge, full description, route links | Cannot claim if tsc-only verified |
| partial | "In Progress" badge, verified_scope summary, honest gap list | Cannot claim "working" or "live" |
| draft | "In Development" badge, description only | Cannot show as functional |
| static_prototype | "Demo / Prototype" badge | Cannot claim as active project |
| unverified | "Coming Soon" if at all | Must not appear with any feature claims |
| planned | "Planned" if at all | Must not appear as existing |

---

## Required Portfolio sections

The first Portfolio version must display:

1. What this is
   — A plain-language explanation of the TLC 2.0 ecosystem and this repo's role.

2. What is working
   — Modules with truth_status=working. Show verified_scope.

3. What is partial
   — Modules with truth_status=partial. Show verified_scope and a short gap summary.

4. What is draft
   — Modules with truth_status=draft. Description only.

5. What is static prototype
   — Exhibits with truth_status=static_prototype. Demo link if available.

6. Evidence
   — Links to artifacts with truth_status=partial or working.

7. What remains unverified
   — Explicit section listing unverified_scope items for partial modules.

8. Governance Core projects
   — Modules with surface=governance_core.

9. Private Lab modules
   — Modules with surface=private_lab. Label as internal/research.

10. Public Exhibits
    — Modules with surface=exhibit. Hackathon submissions, demos.

---

## Forbidden Portfolio patterns

These patterns are forbidden regardless of marketing intent:

- Showing a partial module as "deployed" or "live" with no live_url
- Listing unverified external integrations (Auth0, Gemini, vault) as features
- Claiming "production ready" for any module with unverified_scope items
- Showing mock executors as real execution capability
- Listing a test suite as "passing" when it fails due to a config gap
- Presenting a static HTML file as an interactive application

## Portfolio generation workflow

Step 1: Run the full ingest pipeline on main:
    npm run ingest:verify

This runs:
    npm run verify:registry
    npm run generate:module-status
    npm run generate:portfolio-data

Step 2: Review PORTFOLIO_DATA.json for any field that would overclaim.
    Check every module with public_display_status != hidden.
    Confirm verified_scope_summary is accurate.
    Confirm unverified_scope_summary is honest.

Step 3: Build the Portfolio from PORTFOLIO_DATA.json.
    The Portfolio build process must read PORTFOLIO_DATA.json as its data source.
    No hardcoded module descriptions.
    No hardcoded status claims.

Step 4: Commit PORTFOLIO_DATA.json and MODULE_STATUS.md alongside Portfolio changes.
    The Portfolio and its data source must be in sync at every commit.

## Hyperagent Folio 001 rule

Hyperagent Folio 001 (the prior static HTML design artifact) must be imported
as an exhibit with truth_status=static_prototype and public_display_status=demo.

It must NOT be used as the Portfolio homepage design.
It must NOT be presented as a working application.
It MUST be labeled as a prototype/demo.

---

## Module classification prerequisite

A module must be classified (have a registry entry with truth_status != unverified)
before it can appear on the Portfolio with any status other than "Coming Soon."

The Portfolio build step must enforce this:
- Read PORTFOLIO_DATA.json
- Any module with truth_status=unverified is excluded or shown as "Coming Soon" only
- Any module with no registry entry is excluded entirely
